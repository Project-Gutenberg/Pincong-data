Apache License[\[1\]]() 是著名的非盈利开源组织 Apache 采用的协议。该协议和 BSD 类似，同样鼓励代码共享和尊重原作者的著作权，同样允许代码修改，再发布 (作为开源或商业软件)。需要满足的条件：

Apache License 也是对商业应用友好的许可。使用者也可以在需要的时候修改代码来满足需要并作为开源或商业产品发布 / 销售。

由于 Apache License 并没有反对你使用他们的代码并且闭源再发售，鸿蒙手机系统由于基于 AOSP，在系统应用层面之上加上了分布式总线等功能，然后闭源发布，在关于中也能看到开源许可协议列表 [\[2\]]()，华为都做到了，其做法并没有违反 Apache License，谷歌是没有对其进行反对的理由的。

重点来了，请注意，AOSP 项目包含了两份 Apache License，分别授权许可 **AOSP 源代码**和**周边产品**的部分。

我们分别去阅读一下这两份稍有不同的许可协议。

**1、AOSP 源代码许可协议**

**这是一份适用于系统几乎所有 AOSP 源代码的许可协议。**

依据 AOSP 官网的许可说明文档 [\[3\]]()，AOSP 大部分源代码

[Apache 许可 2.0 版](https://link.zhihu.com/?target=http%3A//www.apache.org/licenses/LICENSE-2.0) (Apache 2.0) 是 AOSP 的首选许可，大多数 Android 软件都使用 Apache 2.0 许可授权用户使用。尽管该项目尽可能遵循此首选许可，但也存在一些例外需要视具体情况处理。例如，与系统使用的许可不同，Linux 内核补丁程序使用的是 GPLv2 许可，您可以在 [kernel.org](https://link.zhihu.com/?target=http%3A//www.kernel.org/pub/linux/kernel/COPYING) 上找到相关信息。

该许可协议是对几乎所有 AOSP 源代码的一个统一的总协议，例外协议的源代码会单独指出。

**2、AOSP 内容许可协议**

**这是一份适用于 AOSP 除系统源代码以外的周边内容资源的协议**

内容许可协议 [\[4\]]()，也是基于 Apache License 2.0 协议，这部分协议适用的范围是 AOSP 周边内容的协议，比如文档内容、文档中的代码、示例代码 example，并且在许可中有这么一段：

我们非常乐意根据相应条款授权您使用 Android 文档和示例代码，这些条款鼓励您根据需要采用、修改、重复使用、改编或混编此类内容。您可以在自己的作品中随意使用文档内容，除非下文的[限制](https://link.zhihu.com/?target=https%3A//source.android.com/license%23restrictions)部分另有说明。例如，您可以在一本书中引用相应的文字，将部分内容剪切并粘贴到您的博客中，录制成有声读物供视障人士收听，甚至还可以翻译成其他语言。

但就是在内容许可上要注意这一点，这也是最关键最有争议的一个点，然而却被很多人忽略，来**敲黑板注意了**：

各位看官，请注意了，前三条说的都是同一件事情。毫无疑问，前三条说的就是，除了源代码授权以外，有些资源尤其是图片是有品牌和商标权的，需要经过谷歌的授权手续才能使用。你的 Android 手机刚开机时底部就有一 Logo 图片显示的是 Powered by Android，谷歌就是授权这张图片的使用以标明发行手机系统的品牌。

相同的道理也是套用在 GMS 上。GMS 的性质和这张图片几乎一毛一样。都需要谷歌的商业授权才能使用，你可以把它们当成手机系统发行的一个海外销售套餐包。谷歌从来没有反对任何不违反这两份协议的个人和企业去使用这个系统并发行自己的手机，你也可以不买这个套餐包，只是买你手机的人就不知道咋下 App 咋用地图了。因为某些众所周知状况，美国的进出口法规禁止谷歌与华为的合作，那么也只能是**商业部分的合作**被终止，也就是有真金白银的 **GMS 与 Android 这张 Logo 图片的商业使用授权被终止**，而华为使用 AOSP 项目只要**不违反 Apache License 2.0 以及限制条款内容**，谷歌是让你随便用去吧，至于你咋玩谷歌就不管了。注意了，这个不管不是真的什么都不管的，你可以做完全兼容，你也可以在 Android 上开发另一套应用 API 开发体系，但是**唯独不可以做不兼容和部分兼容**。这一条莫说谷歌不能容忍，这一条是连 Android 之父 Andy Rubin 也不能容忍的。鸿蒙自己搞出一层 js(类 React) 开发 API，然后包装包装 apk 变成 hap，但是仍然能用 apk，并不制造不兼容，这就是为什么谷歌不去搞他的原因。
