### 知乎用户 somebody 发表
    
对这个问题的探讨不应该只停留在对不对和应该不应该，对 MIUI13 的逆向工程分析也应该做起来，实在不行根据 SNI 抓网络流量判断也行，有能力就抓个现行

思路差不多就这样：

反诈中心的 API 域名一共没几个，根据 HTTPS 请求头能看到请求的域名（SNI 阻断就这个原理），判定是否和 API 有交流。手机上装个证书，解密 HTTPS 流量，检查具体的 API 交互是什么。不建议在 root 下进行测试不然可能会伪装拒绝工作。

逆向工程主要对有可疑功能（根据网络流量耗电量和 Activity 名判断）的系统级 App 进行。

如果有，挑出具体的流量证据还有相关的软件包，有能力反编译给出具体的功能分析，到时候再讨论是否需要整个 Magisk 插件或者 Xp 插件或者全网抵制 MIUI 的事。

如果这么查还查不到证据就是没有，那就不需要讨论假新闻。

国家反诈中心拒绝在 root 环境下运行很可能是 360 加固壳的需要（没错它用的 360 加固），在 root 环境下可以抓取 dex 反编译，拒绝工作的目的很可能是阻止更多的涉及功能的代码加载到内存中被逆向分析。

对于法律问题，尽量不要对联网的部分作修改或者自行对 API 发请求，不要公开反编译的源码和 API 地址。抓包在业内还是很常见的调试方法，只观察数据内容不做修改不太可能违法。

对 MIUI13 的反诈 feature 进行下解析。

### 1、系统级预警

![](https://images.weserv.nl/?url=https%3A//pica.zhimg.com/v2-412eb6743ce2e257b44ec52b0cf9c697_r.jpg%3Fsource%3D1940ef5c)

推测实现思路是：借助反诈中心、合作商的诈骗号码库，或 00、+ 开头的境外号码触发此功能，给予醒目提示。

### 2、诈骗应用安装拦截

![](https://images.weserv.nl/?url=https%3A//pica.zhimg.com/v2-ab2250178825b61102a8a40029fe3a39_r.jpg%3Fsource%3D1940ef5c)

应该是此次争议最大的功能了。miui，包括友商都有类似功能，只不过在 13 这次做得更绝，从用户交互层面上看似没有允许放行的选项（此前 12.5，以及更早的系统，是通过 “纯净模式” 或是在软件包安装程序中可以放行，只不过步骤有些绕）。但也正是如此，也成功保护了父母等不太了解手机的用户，在真正遭受诈骗时的风险。这个是超越反诈中心的功能，毕竟人家一个 app 怎么能干涉你装其他 app。

实现思路：应该是依据此前积累的对疑似风险类 app 的包名、签名之类的特征进行判断，若触发即不允许安装。（此前是阻挠安装，但最终可以装上），这次你也可以装上，只不过要稍微懂点手机，断网。也就避免了无脑想要装上的用户受到风险。

引发大家争议的，是这次把疑似 app，用 “诈骗 app” 一言以蔽之，引发了误会，并以法律法规为由，让用户认为自己遭到了监控。
