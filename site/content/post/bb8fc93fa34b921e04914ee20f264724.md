### 知乎用户  rootkitchao​ 发表
    
理论上会，因为你的上级网络设备能记录你的流量方向，但是你能避免暴露访问内容。

直接访问
----

比如你在浏览器里试图访问 [http://www.google.com](https://link.zhihu.com/?target=http%3A//www.google.com)，浏览器会先检查系统本地的 hosts 文件和 [DNS 缓存](https://zhida.zhihu.com/search?content_id=740357674&content_type=Answer&match_order=1&q=DNS%E7%BC%93%E5%AD%98&zhida_source=entity)，都没有找到这个域名相关的记录，随后浏览器开始向系统网络配置中的 [DNS 服务器](https://zhida.zhihu.com/search?content_id=740357674&content_type=Answer&match_order=1&q=DNS%E6%9C%8D%E5%8A%A1%E5%99%A8&zhida_source=entity)请求解析 [http://www.google.com](https://link.zhihu.com/?target=http%3A//www.google.com)，对方返回了一个地址 142.251.215.228，随后浏览器试图和 142.251.215.228 建立 TCP 连接，数据包在传出你家的路由器后进入运营商的接入网，随后经城域网被转发至骨干网，本来这个数据包应该被发送至北京 / 上海 / 广州的国际互联网出口，但是因为神秘力量的缘故，其所在的 ASN（AS15169,Google LLC）被 [BGP](https://zhida.zhihu.com/search?content_id=740357674&content_type=Answer&match_order=1&q=BGP&zhida_source=entity) 广播为不可达，所以该数据包直接被丢弃，连接建立失败。

Proxy Site+HTTP
---------------

当然聪明的你不会这么容易气馁，于是你买了个一台 [AWS LightSail](https://zhida.zhihu.com/search?content_id=740357674&content_type=Answer&match_order=1&q=AWS+LightSail&zhida_source=entity) 的云服务器，随后又购买了一个域名 [http://1e100proxy.com](https://link.zhihu.com/?target=http%3A//1e100proxy.com) 并为其添加了一条 A 记录 54.236.18.71（你的云服务器 IP），随后在云服务器上使用 [nginx](https://zhida.zhihu.com/search?content_id=740357674&content_type=Answer&match_order=1&q=nginx&zhida_source=entity) 配置了到 google 的反代。随后你用浏览器打开 [http://1e100proxy.com](https://link.zhihu.com/?target=http%3A//1e100proxy.com)，这一次一路顺利，你的电脑和 54.236.18.71 建立了 TCP 连接，并搜索了关键词 T\*\*yoH\*\*。

你的运营商镜像了你的流量，作为中间人它可以看到：

*   你向某个公共 DNS 服务器请求解析 [http://1e100proxy.com](https://link.zhihu.com/?target=http%3A//1e100proxy.com) 并得到结果 54.236.18.71（DNS 协议透明）
*   你通过 HTTP 协议的 GET 方法访问了 [http://1e100proxy.com/search?q=T](https://link.zhihu.com/?target=http%3A//1e100proxy.com/search%3Fq%3DT)\*\*yoH\*\*，以及对方访问的结果

后来中间人根据返回结果中的关键词分析发现了 [http://1e100proxy.com](https://link.zhihu.com/?target=http%3A//1e100proxy.com) 是一个 Google 的 proxy site，于是将其 IP 加入了黑名单，导致你在下一次尝试通过 TCP 协议建立连接时收到了意外的 RST 包无法访问。

很显然直接通过 HTTP 访问并不是一个好主意，对中间人来说其完全透明。

使用 HTTPS
--------

你对中间人能看到你访问的内容和返回结果非常不爽，在搜索一番后你从 AWS 获得了一个新的 IP（54.18.4.102），随后你修改了域名的 A 记录，并且为你的服务器配置了 [TLS 证书](https://zhida.zhihu.com/search?content_id=740357674&content_type=Answer&match_order=1&q=TLS%E8%AF%81%E4%B9%A6&zhida_source=entity)。之后中间人看到的信息是：

*   你向某个公共 DNS 服务器请求解析 [http://1e100proxy.com](https://link.zhihu.com/?target=http%3A//1e100proxy.com) 并得到结果 54.18.4.102
*   你通过 TCP 协议和 54.18.4.102 建立了 TLS 链接，其 [SNI](https://zhida.zhihu.com/search?content_id=740357674&content_type=Answer&match_order=1&q=SNI&zhida_source=entity) 显示访问的域名是 [http://1e100proxy.com](https://link.zhihu.com/?target=http%3A//1e100proxy.com)
*   至于你和该服务器的通信因为 TLS 的双向保密性，中间人无法得知

实际上你访问大多数网站（无论海内外）都是这样，TLS 使得中间人无法得知传输内容，但是 DNS 请求过程和 TLS 的 SNI（Server Name Indication）信息仍然会泄露你访问的域名（仅含域名部分，不包含 GET/POST 的具体内容）。

使用隧道
----

Proxy Site 的灵活性还是欠佳，于是你使用[加密隧道](https://zhida.zhihu.com/search?content_id=740357674&content_type=Answer&match_order=1&q=%E5%8A%A0%E5%AF%86%E9%9A%A7%E9%81%93&zhida_source=entity)转发了你的流量，通过本机 / 路由器上的神秘软件你建立了一条加密隧道。随后你访问 [http://www.google.com](https://link.zhihu.com/?target=http%3A//www.google.com) 时和目标服务器（142.251.215.228）的通信过程全部被转发到你的隧道服务器（198.181.58.212）。此时中间人可以看到你和隧道服务器（198.181.57.140）进行了加密数据传输，然而你向公共 DNS 请求解析 [http://www.google.com](https://link.zhihu.com/?target=http%3A//www.google.com) 的过程还是暴露了你的访问内容，于是你将 DNS 请求也由隧道服务器转发至国外的公共 DNS，避免了暴露。

此时中间人只能看到你和隧道服务器的加密通信，但是完全不知道内容，只能根据特征进行推测。

使用私有信道
------

如果要彻底隐藏流量，则需要使用私有信道，最简单粗暴的方式是用卫星上网，然而星链并未开通服务；另一种方法是使用 IPLC 租用运营商的物理链路，价格大概在数百元每 Mbps 每月（不含两端的服务器）。

这个时候就很好理解为什么说上网的安全与不安全了，做为信息数据传输这个无疑是非常安全的。但对于处处留痕的网路脉络这就不见得是一件安全的事情。

机智的人就已经发现怎么样在网络上不暴露太多个人信息了，就是与自身隐私密切相关的东西不暴露。
