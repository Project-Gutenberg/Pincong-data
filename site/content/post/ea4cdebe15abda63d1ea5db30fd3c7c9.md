当时华为消费者业务总裁余承东发朋友圈说最快今年秋天最晚明年春天 (即刚过完的 2020 年春)，华为自己的操作系统可能面世。而后来鸿蒙却一直难产，以及交货和预期不符等原因，被盼得脖子都长的网友们贴上 PPT 操作系统的标签也就可以理解了。

余承东此翻话一出，当时互联网江湖就一片哗然，然后不少人提出了各方面的质疑，比较集中的应该就是以下提到的几点。

第一，如今手机端操作系统已经是苹果和谷歌两分天下的局势，连做操作系统起家的微软都铩羽而归，华为凭什么认为它能虎口夺食，硬生生的从人家已经切好的蛋糕中再切一块给自己呢？

要做到这一点，那根据《定位》理论，华为必须要树立起一面崭新的旗帜，打出自己的差异化，重新定位对手。后来我们看到了这面大旗上写着的是：鸿蒙是基于微内核的全场景分布式操作系统。

稍微解析下，这里的微内核是相当于宏内核和混合内核说的，也就是内核中只有最基本的 CPU 调度，进程间通讯，中断处理等，将文件系统和硬件驱动等挪到用户层。这样做的好处是灵活，比如有新设备出来了不需要等内核的版本更新才能使用。

大家如果有用过 Linux/Ubuntu 下的 PPA 安装就很好明白了，通常我们如果 apt-get install 找不到对应软件，往往是因为该软件太新了，Ubuntu 还没有将其合进来，而等 Ubuntu 觉得你这个软件没有问题而合进来时，往往是几个月甚至一两年后的事情了，驱动模块加入到 Linux 内核也是类似的。

至于全场景分布式，我之前发了一个视频《秦始皇是如何耗时 10 年搭建起自己的分布式 OS 的》中有通过直白的例子做过详细描述，这里就不多说了，大家感兴趣的可以去看下。

第二个当时大家吐槽的是方舟编译器在做无用功。华为当时宣称自家的方舟编译器可以让操作系统流畅度提升 24%，系统响应速度提升 44%，让重新编译后的安卓应用流畅度可提升 60%，全过程执行机器码而不再是解析运行，彻底解决安卓应用运行慢的问题。也就是说，鸿蒙不会使用 Java 虚拟机，而是开发者打包的时候把所有的静态和动态的库都打包进来作为机器码运行。

![](https://images.weserv.nl/?url=https%3A//pic1.zhimg.com/50/v2-f0284f168897cf5faad96c0edd3edf97_hd.jpg%3Fsource%3D1940ef5c)

但问题是，安卓通过这几年的 JIT 到 AOT 编译技术的发展，加上现在硬件性能越来越强也越来越便宜，软件的运行速度早就不是什么大问题了。那么你大费周折的把反应速度提升个零点几秒的意义何在，用户肉眼根本看不出来，也不会在意。

第三，如果按照上面方舟的做法，那么自然带来另外一个质疑，就是你如果编译成机器码，那跨平台怎么办？当时方舟的一个解决方案是，当开发者打包应用时，方舟同时为不同的硬件平台生成一份机器码，等用户在安装的时候自动选择对应的机器码安装进手机。如果这样做的话，那你一个应用打包肯定远远大于安卓对应的 apk。因为安卓是使用虚拟机解析器来解决跨平台问题的，我一份 java 字节码就可以各个安卓手机平台上跑了。
